

シート1: 全体構成
シート名: 全体構成

1. Djangoの基本アーキテクチャ (MTVモデル)
Djangoは「MTV (Model-Template-View)」と呼ばれるデザインパターンを採用しています。

コンポーネント	名称	役割	一般的なMVCとの対応
M	Model (モデル)	データベースとの連携、データ構造の定義、ビジネスロジックの一部	Model
T	Template (テンプレート)	ユーザーに見せる画面（HTML）の生成、表示ロジック	View
V	View (ビュー)	リクエストを受け取り、モデルとテンプレートを制御してレスポンスを返す司令塔	Controller
2. リクエスト処理の流れ
Webブラウザからアクセスがあった際、Django内部では以下の順序で処理が行われます。

Request: ユーザーがURLにアクセスする。
URLconf (ルーティング): URLを解析し、適切な「ビュー」へ振り分ける。
View (ビュー): 必要な処理を行う（データの取得、計算など）。
Model (モデル): データベースからデータを取得・保存する（必要に応じて）。
Template (テンプレート): データを埋め込んだHTMLを生成する。
Response: 生成されたページをユーザーのブラウザに返す。
3. Djangoの特徴
Batteries Included (バッテリー同梱): 認証、管理画面、サイトマップなど、Web開発に必要な機能が標準で揃っている。
セキュリティ: SQLインジェクションやXSS（クロスサイトスクリプティング）などの脆弱性を防ぐ仕組みが組み込まれている。


シート2: ルーティング
シート名: ルーティング

1. 概要
役割: クライアントからのリクエストURL（例: /products/1/）を見て、どの「ビュー」を実行するかを決定する「案内係」です。
主要ファイル: urls.py
2. 主な機能
URLパターンの定義: path() 関数を使って、URL文字列とビュー関数を紐付けます。
変数の取得: URLの一部を変数として取り出し、ビューに渡すことができます（例: 商品IDなど）。
例: path('products/<int:pk>/', ...) → URLの数字部分を pk としてビューに渡す。
分割管理 (include): プロジェクト全体の urls.py と、各アプリケーションごとの urls.py に分けて管理することで、大規模開発でも整理しやすくなります。
3. 設定例 (イメージ)
URLパターン	呼び出されるビュー	説明
admin/	admin.site.urls	Django標準の管理画面へ誘導
products/	views.product_list	商品一覧画面を表示
products/<int:id>/	views.product_detail	特定の商品詳細画面を表示


シート3: ビュー
シート名: ビュー

1. 概要
役割: アプリケーションの「司令塔」です。リクエストを受け取り、モデルを使ってデータを操作し、テンプレートを使って画面を作成して返します。
主要ファイル: views.py
2. 処理の内容
入力の検証: リクエストパラメータ（GET/POSTデータ）を受け取る。
ビジネスロジック: データの計算、加工、外部APIの呼び出しなどを行う。
データ操作: モデルを通じてデータベースへのクエリ（検索・保存・削除）を実行する。
レスポンス生成: テンプレートにデータを渡してHTMLをレンダリング（描画）し、HTTPレスポンスとして返す。
3. 実装スタイル
Djangoには2つの書き方があります。

スタイル	特徴	向いている用途
関数ベースビュー (FBV)	Pythonの関数として記述。シンプルで処理の流れが見えやすい。	複雑なロジック、小規模な処理
クラスベースビュー (CBV)	Pythonのクラスとして記述。Djangoが用意した汎用クラスを継承して使う。	一覧表示、詳細表示、登録フォームなど、定型的な処理


シート4: モデル
シート名: モデル

1. 概要
役割: データベースのテーブル定義と、データに対する操作（クエリ）を担当します。
主要ファイル: models.py
ORM (Object-Relational Mapping): SQLを書かずにPythonコードでデータベース操作ができる仕組みを提供します。
2. 主な機能
フィールド定義: データの型（文字、数値、日付など）や制約（必須、一意など）を定義します。
例: name = models.CharField(max_length=100)
マイグレーション: モデルの定義変更（Pythonコード）を検知し、自動的にデータベースのテーブル構造を変更（SQL発行）する機能です。
バリデーション: データ保存前に、ルール（最大文字数、形式チェックなど）を満たしているか検証します。
※詳細は別途バリデーション資料を参照
3. 定義とDBの対応
Django (Python)	Database (SQL)
Modelクラス (例: class User(...))	テーブル (例: app_user)
フィールド (例: name, age)	カラム (例: name, age)
インスタンス (例: user1)	レコード (行)


シート5: テンプレート
シート名: テンプレート

1. 概要
役割: ユーザーに表示する画面（HTML）の構造を定義します。ビューから渡されたデータを動的に埋め込むことができます。
格納場所: templates/ ディレクトリ
2. DTL (Django Template Language)
HTMLの中に独自の記法を混ぜて記述します。

記法	名称	役割	例
{{ variable }}	変数	ビューから渡された値を表示する	{{ user.name }}
{% tag %}	タグ	条件分岐や繰り返しなどのロジック	{% if user.is_active %}, {% for item in items %}
`{{ value\	filter }}`	フィルタ	表示する値を加工する
3. テンプレート継承
概要: サイト共通のデザイン（ヘッダー、フッター、ナビゲーションなど）を「親テンプレート」として作成し、各ページでそれを継承して中身だけを記述する機能です。
メリット:
HTMLの重複を排除できる。
デザイン変更時に1ファイルを修正するだけで全ページに反映される。


シート6: フォーム
シート名: フォーム

1. 概要
役割: ユーザーからの入力を受け付け、そのデータが正しいか検証（バリデーション）し、安全なデータとしてプログラムに渡すための仕組みです。
主な機能:
HTML生成: <input> や <select> などのフォームタグを自動生成する。
データ検証: 送信されたデータがルール（必須、型、最大長など）を守っているかチェックする。
データ整形: 検証を通過したデータを、Pythonで扱いやすい型（日付型、数値型など）に変換して cleaned_data に格納する。
2. フォームの種類
Djangoには主に2つのフォームクラスがあります。

クラス名	特徴	向いている用途
forms.Form	汎用的なフォーム。フィールドを1から定義する。	検索フォーム、お問い合わせ、CSVアップロードなど、DB保存と直結しない処理。
forms.ModelForm	定義済みのモデル（Model）を元に自動生成されるフォーム。	ユーザー登録、記事投稿、設定変更など、DBへの保存・更新を行う処理。
3. 基本的な使い方（フロー）
定義: forms.py にフォームクラスを作成する。
表示 (GET): ビューでフォームをインスタンス化し、テンプレートに渡して表示する。
受信 (POST): ビューで request.POST を渡してフォームをインスタンス化する。
検証: form.is_valid() を呼び出す。
成功: form.save() で保存したり、form.cleaned_data からデータを取り出して処理する。
失敗: エラーメッセージ付きで再度フォームを表示する。


シート7: バリデーション
シート名: バリデーション

1. 概要
定義: 入力データや状態が、アプリケーションのルール（仕様）を満たしているかを検証する処理です。
目的: 不正なデータの保存防止、セキュリティの向上、システムの整合性維持。
2. 検証の3つのレイヤー（多層防御）
信頼性を高めるため、複数の場所でチェックを行います。

レイヤー	実施場所	役割・特徴
クライアント側	HTML属性 (required等), JavaScript	ユーザー体験（UX）の向上。即座にミスを指摘するが、セキュリティ的な強制力はない。
サーバー側	Django (Form / Model)	本命の検証。業務ロジックや複雑なルールを適用する信頼できるチェック。
DB側	データベース制約 (UNIQUE, NOT NULL)	最終防衛ライン。アプリのバグですり抜けた不正データも物理的に拒否する。
3. Djangoでの実装方法
検証したい内容に合わせて、実装する場所を使い分けます。

方法	実装場所	向いている用途	例
フィールドオプション	Model / Form定義	基本的な制約	max_length=100, required=True
バリデータ	validators=[...]	再利用可能な単一項目のルール	「0以上の数値」「郵便番号形式」など
clean_()	Formクラス	画面入力に特化した個別ルール	「このメールアドレスは登録不可ドメインです」
clean()	Model / Formクラス	相関チェック（複数項目の整合性）	「終了日時は開始日時より後にする」
4. ベストプラクティス
Defense in Depth (多層防御): クライアント側のチェックだけで満足せず、必ずサーバー側・DB側でも検証する。
エラーメッセージ: 「何がダメで、どう直せばいいか」をユーザーに分かりやすく伝える。
ロジックの集約: データの整合性に関するコアなルールは、可能な限り Model や validators に記述し、どこから保存してもルールが適用されるようにする。
