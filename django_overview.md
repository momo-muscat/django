# Django 概要説明資料

## 1: 概要

### Djangoの設計思想
- Djangoは **MTV（Model / Template / View）** パターンで構成される。
- 「設定より規約」に近い考え方で、標準機能を組み合わせて開発を進める。
- 認証・管理画面・ORM・フォームなど、Web開発の基本機能が最初から揃っている。

### MTVの役割
| 要素 | 役割 | MVCでの近い役割 |
| :--- | :--- | :--- |
| Model | データ構造・DB操作・業務ルールを担当 | Model |
| Template | 画面表示（HTML）を担当 | View |
| View | リクエスト処理と制御（Model/Template連携）を担当 | Controller |

### リクエストからレスポンスまで
1. ブラウザがURLへアクセス
2. URLconfがURLを解析してViewへ振り分け
3. Viewが必要に応じてModelを呼び出し
4. TemplateでHTMLを生成
5. レスポンスとして返却

---

## 2: ルーティング

### 役割
- URLとViewを紐づける「入口制御」。
- どのURLで、どの処理を呼び出すかを明示する。

### 主なポイント
- `path()` でURLパターンを定義する。
- URLパラメータを変数としてViewに渡せる（例: `<int:id>`）。
- `include()` でアプリ単位にURL設定を分割できる。

### 例
| URL | 呼び出し先 | 用途 |
| :--- | :--- | :--- |
| `admin/` | `admin.site.urls` | 管理画面 |
| `articles/` | `views.article_list` | 一覧表示 |
| `articles/<int:id>/` | `views.article_detail` | 詳細表示 |

---

## 3: ビュー / モデル

### Viewの役割
- リクエストを受け取り、処理を判断してレスポンスを返す。
- Formで入力を検証し、Modelでデータ取得・更新を行う。
- テンプレートへ表示データ（context）を渡す。

### Modelの役割
- DBテーブル定義をPythonクラスとして表現する。
- ORMでSQLを意識せずにCRUDを実行できる。
- 制約・バリデーション・業務ルールの中核を保持する。

### 実装パターン
| 分類 | 特徴 | 向いている場面 |
| :--- | :--- | :--- |
| 関数ベースビュー（FBV） | 処理の流れが分かりやすい | 個別ロジックが多い画面 |
| クラスベースビュー（CBV） | 共通処理を再利用しやすい | 一覧/詳細/作成など定型画面 |

---

## 4: テンプレート

### 役割
- Viewから受け取ったデータをHTMLとして描画する。
- 見た目（表示）と処理（Python）を分離し、保守性を上げる。

### DTL（Django Template Language）
| 記法 | 用途 | 例 |
| :--- | :--- | :--- |
| `{{ variable }}` | 値の表示 | `{{ user.username }}` |
| `{% ... %}` | 制御構文 | `{% if %}`, `{% for %}` |
| `|filter` | 表示加工 | `{{ created_at|date:"Y-m-d" }}` |

### テンプレート継承
- `base.html` に共通レイアウトを定義。
- 各画面は必要なブロックのみ上書きする。
- 重複を減らし、デザイン変更を一括で反映できる。

---

## 5: フォーム

### 役割
- 入力UIの生成、入力値チェック、エラー表示を一元化する。
- 安全な入力処理（型変換・クリーニング）を提供する。

### フォームの種類
| 種類 | 特徴 | 代表用途 |
| :--- | :--- | :--- |
| `forms.Form` | 手動で項目定義する汎用フォーム | 検索・問い合わせ |
| `forms.ModelForm` | Model定義から自動生成 | 登録・更新画面 |

### 基本フロー
1. GETで空フォームを表示
2. POSTでフォームへ入力値をバインド
3. `is_valid()` で検証
4. 成功なら保存・処理、失敗ならエラー再表示

---

## 6: バリデーション

### 目的
- 不正データ保存の防止
- セキュリティとデータ整合性の担保
- ユーザーに分かりやすい入力フィードバックを提供

### 多層防御の考え方
| 層 | 主な実施場所 | 役割 |
| :--- | :--- | :--- |
| クライアント層 | HTML属性 / JavaScript | 早期フィードバック（UX向上） |
| サーバー層 | Django Form / Model | 信頼できる本命チェック |
| DB層 | UNIQUE / NOT NULL など | 最終防衛ライン |

### Djangoでの実装ポイント
- フィールド制約（`max_length`, `blank`, `unique`）
- `validators` で再利用可能なルール化
- `clean_<field>()` で項目単位のチェック
- `clean()` で項目間の相関チェック

### まとめ
- バリデーションは「入力時」だけでなく「保存時」まで意識する。
- ルールは可能な限りModel/Formに集約し、再利用性を高める。
