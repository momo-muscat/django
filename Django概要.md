# Django概要（ブレークダウン版）

Excelで各シートに貼り付けやすいよう、各テーマを「役割」「何をするか」「どう実装するか」に分けて整理しています。

## シート1: 全体構成

### 1. 目的
- Django全体の処理構造（MTV）を理解する。
- リクエストがどの順番で処理されるかを把握する。

### 2. MTVモデルの分解

| コンポーネント | 名称 | 主な責務 | 一般的なMVCとの対応 |
|---|---|---|---|
| M | Model (モデル) | DB構造の定義、データ取得・保存、一部ビジネスルール | Model |
| T | Template (テンプレート) | 画面表示用HTMLの組み立て、表示ロジック | View |
| V | View (ビュー) | リクエスト受付、Model/Templateの呼び出し制御、レスポンス返却 | Controller |

### 3. リクエスト処理の分解
1. `Request`: ユーザーがURLへアクセス。
2. `URLconf`: URLパターンに一致するビューを特定。
3. `View`: 必要な処理を実行（入力処理、データ取得など）。
4. `Model`: DBとやり取り（必要時）。
5. `Template`: 画面表示用HTMLを生成。
6. `Response`: ブラウザへ結果を返却。

### 4. 特徴（導入で伝えるポイント）
- Batteries Included: 認証、管理画面、サイトマップなどが標準搭載。
- セキュリティ機能: SQLインジェクション/XSS対策などがフレームワーク側に組み込まれている。

## シート2: ルーティング

### 1. 役割
- URLとViewを対応付ける窓口。
- 主要ファイル: `urls.py`

### 2. 入力・出力で見る
- 入力: ブラウザからのURL（例: `/products/1/`）
- 出力: 実行されるビュー関数/クラス

### 3. 主要機能の分解
- `path()` でURLパターンを定義する。
- URLの可変部分をパラメータとして受け渡す。
- `include()` でURL設定をアプリ単位に分割する。

### 4. 設定例

| URLパターン | 呼び出されるビュー | 用途 |
|---|---|---|
| `admin/` | `admin.site.urls` | 管理画面 |
| `products/` | `views.product_list` | 一覧表示 |
| `products/<int:id>/` | `views.product_detail` | 詳細表示 |

### 5. Excel転記用の一言
- 「URLを見て、どの処理を動かすか決める層」

## シート3: ビュー

### 1. 役割
- アプリケーションの司令塔。
- 主要ファイル: `views.py`

### 2. 処理をステップ分解
1. リクエストデータを受け取る（GET/POST）。
2. 必要な業務ロジックを実行する。
3. Model経由でDB操作する。
4. Templateへデータを渡し、レスポンスを生成する。

### 3. 実装スタイル比較

| スタイル | 特徴 | 向いているケース |
|---|---|---|
| 関数ベースビュー (FBV) | 1つの関数に処理を書く。流れが追いやすい。 | 小〜中規模、細かい制御が必要な処理 |
| クラスベースビュー (CBV) | Django汎用クラスを継承して構築する。 | 一覧/詳細/作成/更新などの定型画面 |

### 4. 最小イメージ
```python
def product_list(request):
    products = Product.objects.all()
    return render(request, "products/list.html", {"products": products})
```

## シート4: モデル

### 1. 役割
- DBテーブル定義とデータ操作を担当する。
- 主要ファイル: `models.py`

### 2. できることの分解
- フィールドでデータ型と制約を定義。
- ORMでPythonコードからDB操作。
- マイグレーションでDB構造を更新。
- 保存前バリデーションで不正値を防止。

### 3. 定義とDBの対応

| Django側 | DB側 |
|---|---|
| Modelクラス | テーブル |
| フィールド | カラム |
| インスタンス | レコード（行） |

### 4. 最小イメージ
```python
class Product(models.Model):
    name = models.CharField(max_length=100)
```

## シート5: テンプレート

### 1. 役割
- 画面表示（HTML）を担当する。
- 主要ディレクトリ: `templates/`

### 2. DTLの要素分解

| 記法 | 用途 | 例 |
|---|---|---|
| `{{ ... }}` | 値を表示 | `{{ user.name }}` |
| `{% ... %}` | 制御構文（if/for等） | `{% if user.is_active %}` |
| `{{ value|filter }}` | 値を加工して表示 | `{{ created_at|date:"Y-m-d" }}` |

### 3. テンプレート継承
- 共通レイアウトを親テンプレートに集約。
- 子テンプレートはページ固有部分だけを記述。
- 重複削減と保守性向上に効果的。

## シート6: フォーム

### 1. 役割
- ユーザー入力の受け取り、検証、整形を担う。

### 2. 機能の分解
- HTMLフォーム部品を生成する。
- 送信値を検証する（必須、型、長さなど）。
- 正常値を `cleaned_data` として扱いやすい型に変換する。

### 3. フォーム種類の比較

| クラス | 特徴 | 向いている用途 |
|---|---|---|
| `forms.Form` | フィールドを手動定義 | 問い合わせ、検索などDB非依存処理 |
| `forms.ModelForm` | Model定義から自動生成 | 登録・更新などDB保存を伴う処理 |

### 4. 基本フロー
1. `forms.py` でフォーム定義。
2. GETで空フォームを表示。
3. POSTで `request.POST` を渡す。
4. `is_valid()` で検証。
5. 成功時は保存または処理。
6. 失敗時はエラー付き再表示。

## シート7: バリデーション

### 1. 目的
- 仕様に合わないデータを保存させない。
- セキュリティとデータ整合性を守る。

### 2. 3層防御の分解

| レイヤー | 実施場所 | 役割 |
|---|---|---|
| クライアント側 | HTML属性/JavaScript | UX向上、即時チェック |
| サーバー側 | Django Form/Model | 信頼できる本命チェック |
| DB側 | `UNIQUE`/`NOT NULL` 等 | 最終防衛ライン |

### 3. Djangoでの実装ポイント

| 方法 | 実装場所 | 使いどころ | 例 |
|---|---|---|---|
| フィールドオプション | Model/Form定義 | 基本制約 | `max_length=100` |
| バリデータ | `validators=[...]` | 再利用したい単項目ルール | 数値範囲、形式チェック |
| `clean_<field>()` | Form | 画面入力に依存した個別ルール | 特定ドメインのメール拒否 |
| `clean()` | Form/Model | 項目間の整合性チェック | 開始日 < 終了日 |

### 4. ベストプラクティス
- クライアント側だけに依存せず、必ずサーバー側とDB側でも検証する。
- エラーメッセージは「何が問題か」と「どう直すか」を明確にする。
- コアな整合性ルールはModel/validatorへ寄せ、再利用性を高める。
